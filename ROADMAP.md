# MCP-PM 版本路线图

> **项目**: mcp-pm (Model Context Protocol Package Manager)
> **定位**: "Homebrew for MCP Servers" — 统一的 MCP 服务器包管理器
> **技术栈**: Python 3.11+ · Click · FastAPI · httpx · Pydantic
> **作者**: Davey Wong <wgwcko@gmail.com>

---

## 总览

```
v0.1.0 ──→ v0.2.0 ──→ v0.3.0 ──→ v0.4.0 ──→ v0.5.0 ──→ v1.0.0
  MVP       HTTP代理      Web UI     安全沙箱    插件系统    生产就绪
```

每个版本预计开发周期 4–6 周，版本间有 1–2 周缓冲期用于 bug 修复和社区反馈。

---

## v0.1.0 — MVP (Minimum Viable Product)

**目标**: 实现最核心的 CLI 包管理功能，让用户能安装、搜索、列出 MCP 服务器。

### 核心功能

| 功能 | 模块 | 描述 |
|------|------|------|
| `mcp-pm install <name>` | `installer.py` | 从注册中心安装 MCP 服务器，支持指定版本 |
| `mcp-pm uninstall <name>` | `installer.py` | 卸载已安装的 MCP 服务器 |
| `mcp-pm list` | `config.py` | 列出所有已安装的 MCP 服务器 |
| `mcp-pm search <query>` | `registry.py` | 搜索注册中心中的 MCP 服务器包 |
| `mcp-pm info <name>` | `registry.py` | 查看包的详细信息 |
| `mcp-pm config` | `config.py` | 查看和编辑本地配置 |
| `mcp-pm doctor` | 多模块 | 诊断系统环境与配置状态 |
| 注册中心客户端 | `registry.py` | 与远端 Registry API 通信（查询/搜索） |
| 配置管理 | `config.py` | YAML 配置文件读写、多 profile 支持 |
| 安装器 | `installer.py` | pip 安装、版本解析、依赖检查 |

### 技术难点

1. **MCP 协议兼容性** — 需要兼容不同 MCP Server 的传输实现（stdio 为主）
2. **跨平台路径处理** — Windows/macOS/Linux 三平台的配置目录和虚拟环境路径
3. **版本解析** — semver 范围匹配与依赖冲突检测
4. **错误处理** — 网络超时、安装失败、配置损坏等场景的优雅处理

### 交付物

- 完整的 CLI 接口（9+ 个子命令）
- 注册中心客户端 SDK
- 本地配置管理系统
- 安装/卸载流水线
- 80%+ 单元测试覆盖率
- 开发者文档 (DEVELOPER_GUIDE.md) + 架构文档 (ARCHITECTURE.md)

### 估计时间: **5 周**

| 周次 | 里程碑 |
|------|--------|
| W1 | 项目脚手架、CLI 框架、配置管理 |
| W2 | 注册中心客户端 + 搜索/查看 |
| W3 | 安装器核心逻辑 |
| W4 | 卸载/列表/诊断 |
| W5 | 测试、文档、发布 |

## v0.2.0 — HTTP 代理服务器

**目标**: 提供 OpenAI API 兼容的 HTTP 代理，让 AI 应用（Cursor、Claude Desktop 等）通过标准 API 调用 MCP 工具。

### 核心功能

| 功能 | 模块 | 描述 |
|------|------|------|
| `POST /v1/chat/completions` | `server.py` | OpenAI 兼容的聊天补全 API（工具调用路由） |
| `POST /v1/tools/list` | `server.py` | 列出所有可用 MCP 工具 |
| `POST /v1/tools/call` | `server.py` | 调用指定 MCP 工具 |
| `GET /v1/models` | `server.py` | 列出可用模型（映射到已安装 MCP 服务器） |
| SSE 流式响应 | `server.py` | Server-Sent Events 实时流输出 |
| `mcp-pm server start` | `cli.py` | 启动/停止/重启 HTTP 代理 |
| `mcp-pm run <name>` | `client.py` | 直接运行 MCP 服务器（stdio 模式） |
| MCP 客户端运行时 | `client.py` | stdio/HTTP 双传输协议支持 |

### 技术难点

1. **OpenAI API 语义映射** — 将 chat/completions 的 `tool_calls` 语义映射到 MCP 协议的 `call_tool`
2. **流式工具调用** — 支持 SSE 流式返回，同时处理工具调用的中间状态
3. **多服务器路由** — 多个 MCP 服务器下的工具名称冲突与路由策略
4. **连接管理** — MCP 服务器进程的生命周期管理、健康检查、自动重启

### 估计时间: **4 周**

| 周次 | 里程碑 |
|------|--------|
| W1 | MCP 客户端运行时 (StdioTransport + HTTPTransport) |
| W2 | FastAPI 代理服务器 + OpenAI 兼容端点 |
| W3 | 流式响应 + 多服务器路由 |
| W4 | CLI server 命令、集成测试、文档 |

## v0.3.0 — Web UI + 工具测试界面

**目标**: 提供图形化的 Web 管理界面，降低使用门槛。同时提供工具测试沙箱，方便开发者调试。

### 核心功能

| 功能 | 模块 | 描述 |
|------|------|------|
| Dashboard 首页 | `webui/` | 服务器概览、状态、安装统计 |
| 服务器管理 | `webui/` | 可视化安装/卸载/更新 |
| 配置编辑器 | `webui/` | Web 版配置编辑 |
| 工具测试沙箱 | `webui/` | 在线调用工具并查看结果 |
| 日志查看器 | `webui/` | 实时日志浏览 |
| `mcp-pm webui` | `cli.py` | 启动 Web Dashboard |
| 工具调用历史 | `client.py` | 记录和回放工具调用 |

### 技术难点

1. **HTMX + Jinja2 前端架构** — 无 SPA 框架下的动态交互实现
2. **实时日志推送** — 使用 SSE 推送后端日志到前端
3. **工具测试沙箱** — 安全地在 Web 界面上执行工具调用，防止 XSS
4. **响应式布局** — 在桌面和移动端均有可用体验

### 估计时间: **5 周**

| 周次 | 里程碑 |
|------|--------|
| W1 | Web 框架搭建、Dashboard 页面 |
| W2 | 服务器 CRUD 操作界面 |
| W3 | 工具测试沙箱 |
| W4 | 配置编辑器 + 日志查看器 |
| W5 | 端到端测试、UI 打磨 |

## v0.4.0 — 安全沙箱 + 权限管理

**目标**: 引入安全隔离机制，让用户放心运行第三方 MCP 服务器。提供细粒度的权限控制。

### 核心功能

| 功能 | 模块 | 描述 |
|------|------|------|
| 虚拟环境隔离 | `sandbox.py` | 为每个服务器创建独立 venv |
| Docker 容器隔离 | `sandbox.py` | Docker 容器级隔离（高安全） |
| 子进程 seccomp | `sandbox.py` | Linux seccomp 系统调用过滤 |
| 资源限制 | `sandbox.py` | CPU / 内存 / 网络 / 磁盘配额 |
| 权限声明 | `config.py` | 每个工具声明所需权限（文件/网络/执行） |
| 权限审批 | `cli.py` + `webui/` | 安装时确认权限，运行时可审计 |
| 网络白名单 | `sandbox.py` | 限制服务器可访问的网络地址 |
| 文件系统隔离 | `sandbox.py` | 只读系统文件 + 限定写入路径 |

### 技术难点

1. **seccomp 策略配置** — 不同 MCP 服务器的系统调用差异，需要自动生成白名单
2. **Docker 集成** — 无 Docker 环境的优雅降级、Docker 镜像管理
3. **权限模型设计** — 权限声明的 DSL 设计（声明式 vs 编程式）
4. **性能开销** — 隔离层带来的延迟，需要在安全和性能间平衡

### 估计时间: **6 周**

| 周次 | 里程碑 |
|------|--------|
| W1 | 沙箱抽象层定义 + venv 策略 |
| W2 | Docker 容器策略（拉取/创建/清理） |
| W3 | seccomp / 资源限制 |
| W4 | 权限声明 DSL + 审批流程 |
| W5 | 网络和文件系统隔离 |
| W6 | 安全审计、渗透测试、文档 |

## v0.5.0 — 插件系统 + 社区贡献

**目标**: 开放插件体系，让社区可以贡献传输协议、沙箱策略、注册中心后端等扩展。

### 核心功能

| 功能 | 模块 | 描述 |
|------|------|------|
| 插件 SDK | `plugins/` | 插件开发工具包（基类、注册机制） |
| 传输协议插件 | `plugins/transport/` | Websocket、gRPC、Unix Socket 等 |
| 沙箱策略插件 | `plugins/sandbox/` | Firecracker、K8s 等沙箱策略 |
| 注册中心后端插件 | `plugins/registry/` | GitHub、npm、自定义注册中心 |
| 输出格式化插件 | `plugins/formatter/` | JSON、表格、自定义输出格式 |
| Hooks 系统 | `installer.py` | 安装前/后 Hook、事件系统 |
| 包发布工具 | `cli.py` | `mcp-pm publish` 发布到注册中心 |
| 注册中心 API | `registry.py` | 包发布、版本管理、下载统计 |

### 技术难点

1. **插件热加载** — 安装后无需重启即可加载插件
2. **插件 API 稳定性** — 需要定义稳定的契约接口，保证向后兼容
3. **依赖管理** — 插件间的依赖关系和版本冲突
4. **权限安全** — 插件运行在何种权限级别，防止恶意插件

### 估计时间: **6 周**

| 周次 | 里程碑 |
|------|--------|
| W1 | 插件 SDK 设计 + 注册机制 |
| W2 | 传输协议插件实现（Websocket、gRPC） |
| W3 | 沙箱/注册中心后端插件 |
| W4 | Hooks 系统 + 事件总线 |
| W5 | 包发布工具 + 注册中心 API 扩展 |
| W6 | 示例插件、社区指南、文档 |

3. **大规模测试** — 数千个 MCP 服务器的安装/运行测试
4. **安全合规** — 满足企业安全需求（审计日志、RBAC、SSO）
